01
Quick Start

A way in. Synthetic signal. No hardware, no scans, no stakes — you walk the corridors of the Grid before you trust it with the real thing. One command. It fires up the whole world, pre-populated.

terminal
# Clone and enter the repo
$ git clone https://github.com/your-org/homelab-dashboard
$ cd homelab-dashboard

# One command — spins up frontend + backend with mock data
$ make mock
◈
make mock lights up the frontend on port 5173 and the backend on port 8000 with BACKEND_MOCK=true. No Nmap. No UniFi. No OpenVAS. Eleven phantom devices, full topology — the Grid, in a bottle.
browser
# Open in your browser
→ http://localhost:5173

Inside your own machine — before any of this hits real hardware — you're in the safe corner of the Grid. Hot reloads. Synthetic data. Every keystroke echoing back within milliseconds. The place where you build, and break, and build again.

Manual Backend Start

If you need to pass custom environment variables, run the backend directly:

terminal
# Create and activate venv (first time only)
$ cd backend
$ python3 -m venv .venv
$ source .venv/bin/activate
$ pip install -r requirements.txt

# Mock mode (no real integrations)
$ BACKEND_MOCK=true DB_PATH=/tmp/homelab.db \
    .venv/bin/uvicorn main:socket_app \
    --host 0.0.0.0 --port 8000 --reload

# Live mode with real scans
$ DB_PATH=/var/data/homelab.db \
    ANTHROPIC_API_KEY=sk-ant-... \
    .venv/bin/uvicorn main:socket_app \
    --host 0.0.0.0 --port 8000

terminal
$ cd frontend
$ npm install
$ npm run dev          # dev server → http://localhost:5173
$ npm run build        # TypeScript check + production build

04
Setup Wizard

First cycle on the Grid. Setup happens in two passes — compose-time and runtime — and each one owns its own territory. The bash wizard asks the bare minimum: where the server lives on the network, how the browser will reach it, whether TLS is in play. Once the stack is up, the in-app wizard opens and handles everything else: UniFi, Elasticsearch, OpenVAS, Claude. Gateway integrations — OPNsense, Firewalla — are opt-in, configurable from the Settings page at any time after the first launch; leave them dark and the Grid still runs on UniFi + Nmap alone. Secrets land in config.yml, not in shell env files. Two wizards, one Grid, no overlap.

◈ Wizard Steps
  1. Network
    Declare the territory. CIDR range the scanner will sweep (e.g. 192.168.1.0/24), the gateway that rules it. This is the map — the Grid needs to know where its edges are.
  2. Elasticsearch
    Long memory. Point to an existing Elasticsearch node (e.g. http://elastic.local:9200) and the dashboard will keep a record of every scan, every finding, every cycle. Test the link before you move on.
  3. UniFi Controller
    The authority. UDM Pro or classic controller — url, credentials, site. Hit Test and the field transforms into a dropdown of real sites. No guessing. No typos. Pick the one that's yours.
  4. Services
    The specialists. OpenVAS for vulnerability sweeps. Claude for the programs you can't identify — fed by ANTHROPIC_API_KEY or the claude CLI. Each one's optional. Each one makes the Grid a little smarter.
  5. Review & Launch
    Last look. Every value, laid out plain. Hit Launch Dashboard and the config drops to disk, the scheduler wakes, the first scan fires within seconds. You're in.
⚠
Flip BACKEND_MOCK=true and the wizard stands down. The backend declares setup_complete: true on boot and drops you straight onto the Grid. Useful for demos. Dangerous for production.

Production Deploy

Time to push the Grid out beyond the workstation. One command, one server, one stack. No ceremonies. No sudo waltzes. The script does the heavy lifting — you watch the lights come on.

Method A — deploy.sh (SSH + Docker Compose)

The clean line. A remote Ubuntu or Debian host, Docker already installed, and your SSH key on file. That's the whole shopping list.

terminal
# Option A — positional argument (recommended)
$ ./deploy.sh ubuntu@192.168.1.50

# Option B — environment variables
$ DEPLOY_HOST=192.168.1.50 DEPLOY_USER=ubuntu ./deploy.sh

# Option C — interactive prompt (no arguments)
$ ./deploy.sh
# → prompts: "Target host (user@ip or DEPLOY_USER@DEPLOY_HOST):"

# The script will:
#  1. Build frontend (npm run build)
#  2. rsync project files to /opt/homelab-dashboard on the target
#  3. Run docker compose up -d --build
#  4. Poll /api/health until the stack reports healthy (60s timeout)
#  5. Print the dashboard URL
Method B — Docker Compose Direct
terminal
# Copy and fill in your environment file
$ cp .env.example .env
$ $EDITOR .env

# Start core services
$ docker compose up -d

# Start with NGINX Proxy Manager
$ docker compose --profile proxy up -d

# Start with OpenVAS scanner
$ docker compose --profile openvas up -d

# Full stack — all optional services
$ docker compose --profile proxy --profile openvas up -d

# Check service health
$ docker compose ps
$ curl http://localhost:8000/api/health

Network Map

The main view. A living topology, every program on your network plotted in a single frame. Two cameras on the same data:

Sphere · the 3D Grid

The landing shot. Devices arrange themselves on a glowing wireframe sphere using a Fibonacci distribution — even angular spacing at any node count, so ten devices look as composed as a hundred. Grab it, drag it, let go — the sphere spins with momentum and coasts to rest. Scroll to zoom in through the shell and out to the full Grid. An idle auto-rotation starts the moment you arrive and stops the moment you touch it.

Every label is billboarded with a horizon lock: the text yaws to face you but never pitches or rolls, so reading stays effortless no matter how the sphere is oriented. Labels on the far side of the sphere fade out so they don't bleed through. Device orbs pulse when selected; hovering scales them up. Critical CVEs paint the orb red, highs paint it orange, otherwise it's the usual online/offline/unknown palette.

The entire 3D engine (three.js + postprocessing, ~287 kB gzipped) lazy-loads on first visit to this page — it never ships with the initial bundle. On GPUs that struggle, a PerformanceMonitor dials the device pixel ratio down automatically; while you're actively orbiting, an AdaptiveDpr drop keeps frames fluid, then snaps back to sharpness when you release.

Flow · the 2D graph

The pragmatic view. React Flow renders nodes you can drag to reposition, with a minimap and pan/zoom controls. The gateway rides at the center of the Grid. Switches and access points orbit close — the infrastructure ring. Endpoints fan out past them in concentric circles. Node positions persist server-side, so your layout survives restarts.

Swap between views with the Sphere / Flow toggle in the top-right corner. Your preference persists to localStorage; the next session lands on whichever camera you left.

07
Devices

The same programs. A different camera angle. Every device in a sortable ledger — filterable, searchable, bulk-reviewable. When you need the audit, not the poetry.

Filter Controls
  • Status filter — all / online / offline / unknown. Narrow the field.
  • Search — IP, hostname, MAC, device type. Whatever you remember, it'll catch.
  • Sort — by IP, by last seen, by CVE count. The noisiest programs rise to the top.
Scan Column

The per-row Scan button — the fastest path from "I know which device I want" to "a scan is running against it." One click posts /api/vulns/scan/{device_id}; the button immediately flips through a little state machine so your click doesn't vanish into the void:

  • queueing → queued → running — a spinning pip on cyan. A toast fires the moment the scan is accepted.
  • done — a green check. A second toast points you at the Vulnerabilities page for results.
  • failed — a red retry button with the error in its tooltip. One more click reruns it.
  • If a device has no IP, the button disables with a tooltip explaining why — column alignment stays consistent.

Completion rides the scan:complete socket event with the device id attached. If the event gets lost (socket reconnect, process restart), a 90-second TTL reverts the row to idle so you're never staring at a forever-spinner.

Device Detail Panel

Click a row. Same slide-in panel as the Map — the view just gets richer here:

  • Services — every open port the scanner found. Protocol, service name, version. Where a program speaks HTTP on a known UI, you get a launch link — one click straight to the console.
  • Vulnerabilities — CVEs filed against this device. Expand a row for the full write-up, the CVSS band, remediation steps. Every CVE ID lands on its NVD entry.
  • Scan — same per-device OpenVAS run as the column button, exposed inside the panel too.

09
Claude Integration

Every so often the scanner meets a stranger — a program with no known signature. The Grid doesn't shrug. It hands the stranger to Claude. Ports, banners, OS guesses — Claude reads them like tea leaves, makes an identification, and writes a Python integration on the spot. You get the last word on whether it's kept.

Analysis Pipeline
  1. Unknown Device Detected
    After every Nmap pass, the unclassifieds get funneled into run_analysis_for_unknown_devices(). Anything already waiting in the review queue is left alone — no duplicates.
  2. Claude Analysis
    The full context — IP, ports, banners, OS guess — goes out as a structured prompt. Claude comes back with a name for the stranger and an integration.py carrying DEVICE_TYPE, DEVICE_LABEL, and an async enrich(ip). A program built to understand programs.
  3. Staged Change Created
    The new file lands in a sandbox, not the live tree. A record drops into the database; claude:staged fires over the socket; the notification banner glows at the top of the dashboard. Nothing's loose yet.
  4. Review in Audit Log
    Head to Audit Log → Change History. Expand the row. You'll see what Claude saw, the diff it wrote (colored up), and every file slated for deployment.
  5. Approve or Reject
    Approve — the integration file graduates out of the sandbox and into INTEGRATIONS_DIR (default ~/homelab-dashboard/integrations/devices/). Reject — marked, and the sandbox is wiped. Either way, the audit log remembers.
Claude CLI vs SDK Fallback
MethodRequirementBehavior
Claude CLI claude in PATH First choice. claude --print --allowedTools Write,Edit,Read runs in the sandbox, full Claude Code context. Sharpest result.
Anthropic SDK ANTHROPIC_API_KEY set Fallback — Messages API, claude-opus-4-7. The runner pulls the first Python block out of the response. Still good.
Placeholder Neither available Last resort. A stub integration.py with TODO markers, still staged. You finish it by hand — the pipeline refuses to drop the stranger on the floor.
Audit Log Tabs
  • Change History — every staged change (pending, approved, rejected, applied). Expandable diff viewer. Device context attached.
  • Audit Log — the immutable ledger. Who did what, when. No edits, no rewrites.
What Gets Logged

The Audit Log started life as a Claude-only ledger but now catches every meaningful UI mutation the Grid makes. Expand any row and the detail block humanises the event: which sections of the settings were touched, which hosts a credential test failed against, which device a scan fired at. Secrets (passwords, API keys, private keys) never make it to the log — only the non-secret fields around them.

ActionFires On
save_settingsIntegration credentials saved via the setup wizard or Settings panel
save_scan_intervalsScheduler intervals changed
save_scan_credential · delete_scan_credentialPer-target SSH credentials added or removed

The log is intentionally capped at 100 rows. Every write trims the oldest entries past the cap in the same transaction, so the ledger stays small and fast without a background job to prune it. If you need forever-history, Elasticsearch is the right destination — the Grid's audit log is designed for the last hundred actions, not the last hundred thousand.

10
Security & Alarms

The Grid watches its own perimeter. Suricata IDS drops on OPNsense. Firewalla MSP cloud alarms. Every gateway that cares to speak up gets its own feed into a single unified surface — the Security page, pinned to the sidebar behind the Bell icon. No separate dashboards, no tab-juggling. One inbox, sorted, deduplicated, live.

The Bell

A quiet sentry in the sidebar. When there's nothing to say, it sits dark and still. When an alarm lands — Suricata rule fires, Firewalla flags a new device, anything the gateway thought worth a siren — a red dot blooms in the upper-right of the icon. The count rides along with it, capped at 99+ so a full-blown alert storm doesn't smear across the chrome. Hover for the exact count; click to walk into the feed.

The indicator is driven by the alarm:summary socket channel — the server pushes the unacknowledged count the moment anything ticks it up or down, and a 60-second poll is the background safety net in case a socket drops. The badge is always honest within a second of the truth.

Severity Bands

Five tiers, Tron-palette, matching the Vulnerabilities page vocabulary so the eye doesn't have to relearn the lights on every page:

SeverityColorWhat It Means
criticalredActive exploit or confirmed compromise. Treat as an incident.
highorangeTargeted malicious traffic. Suricata priority 1, Firewalla security-activity types.
mediumyellowSuspicious but not yet actionable. Port scans, reputation hits, unusual flow volumes.
lowgreenNoise-floor. New device joined, low-confidence signature, informational IDS category.
infoslatePure informational. Box came online, health ping, scheduled event.

Suricata priorities translate cleanly: 1→critical, 2→high, 3→medium, 4→low. Firewalla alarms carry a numeric type code instead of a severity field — the backend maps the catalogue (security activity, intrusion, open port, new device, etc.) onto the same five-tier vocabulary so both sources sort against each other without asterisks.

Source Chips

Every row carries a chip that names who raised it — opnsense (green) for Suricata-fed alerts, firewalla (red) for MSP cloud alarms. The filter at the top of the page narrows the feed by source, by severity, or by acknowledged state. Nothing gets hidden by default — unacknowledged and acknowledged both show, and the toggle to reveal dismissed items is one click.

Deduplication

Gateways are generous with repetition. A single scanning host will spray the same Suricata rule dozens of times a minute; a Firewalla on a flaky ISP will re-fire a New-Device alarm every time the MAC reassociates. The Grid collapses these into a single row with a visible seen × N counter and the most recent timestamp. The fingerprint for the collapse is deterministic:

  • OPNsense / Suricata — src_ip | dst_ip | signature | minute_bucket
  • Firewalla — gid | device_mac | type_code | minute_bucket

Two alarms that match on every dimension within the same minute become one row with an incremented counter. Next minute, new row. Storms stay loud enough to notice; volume never drowns the signal.

Row Actions

Every row is an expandable card. The collapsed view shows severity, source, a one-line message, the device (if one could be pinned), and the relative timestamp. Click to expand and the raw gateway payload unfolds underneath — the full JSON as it came off the wire, pretty-printed for eyeballing. No hidden fields, no "contact support for details." What the gateway said is what you see.

Three actions live on every row:

  • Acknowledge — "I've seen this." The row stays in the feed, the unacked counter ticks down, the sidebar Bell loses one from its badge. Reversible — ack again to unack.
  • Dismiss — "Not worth my time." The row is hidden from the default view but not deleted; flip Include dismissed to audit what got swept.
  • Archive dismissed — the one nuclear option, up by the filter row. Wipes dismissed alarms from the database outright. Logged to the audit trail; no undo.
Live Updates

The feed is real-time. When a gateway raises a new alarm, the backend emits alarm:new on the socket and the page slides the fresh row to the top without a reload. alarm:summary rides alongside and updates both the sidebar badge and the six stat cards at the top of the page (Total, Unacknowledged, Critical, High, Medium, Low) in the same beat. A 60-second React Query poll runs underneath as insurance — if the socket drops, the UI is still honest within a minute.

▸
No gateway configured? The Security page still loads with an empty state — it's always in the nav so the surface is discoverable. Configure OPNsense or Firewalla under Settings → Gateway Integrations and the feed begins populating on the next poll cycle.

Settings
Scheduler Panel

The Grid runs on clocks. Six of them, listed here with their next firing time:

JobDefault IntervalDescription
nmap_scan Every 15 min Walks the configured CIDR, finds every program, updates the topology. Any stranger it turns up gets handed to Claude.
unifi_poll Every 30 sec Pulls the managed client list off the UniFi controller — hostnames, switch ports, AP associations. Puts names to the wire.
latency_poll Every 30 sec TCP-taps the gateway and infrastructure devices, logs the round-trip. Feeds the latency chart on the Network Map. Prunes anything older than 24 hours.
opnsense_poll Every 60 sec Hits the OPNsense REST API for the interface roster, ARP/DHCP leases, and (when IDS is enabled) fresh Suricata alerts. Merges leases into the device table and pushes alarms onto the Security feed. Only registered when an OPNsense credential is configured.
firewalla_poll Every 120 sec Hits the Firewalla MSP cloud API for the box roster, device list, and active alarms. Cursor-paginates until the feed is drained. Same dedup + alarm pipeline as OPNsense. Dormant when Firewalla is disabled or configured in local-only mode.
openvas_scan Every 24 hrs The deep pass. Vulnerability scans against everything online. Findings land in the database and vuln:updated lights up the UI.

Need it now? Hit Run Now on any row. The schedule keeps ticking; the job just gets an early bell.

Need it stopped? Hit Pause. The job disappears from the firing schedule but stays listed with a paused badge — Run Now still works for one-off runs. Resume puts it back on the clock. Pauses are in-memory: a backend restart un-pauses everything, which is deliberate. If you want a scheduled job off permanently, change its interval, don't pause it.

Scan Intervals

Every interval is an input box. Change the number, hit save — the scheduler reprograms itself in place. No restart, no downtime.

Integration Credentials

UniFi, Elasticsearch, OpenVAS — each one gets its own panel. Type in the host, user, password. Hit Test to bounce a real authenticated call at the target — UniFi logs in and returns the site list, Elasticsearch pings its cluster endpoint, OpenVAS opens a GMP session and calls authenticate. A green pill means the Grid can actually use the credentials, not just reach the port. Hit Save to drop the value into config.yml. Leave a password field empty and the stored value stays put — no need to retype what's already on the wire.

OpenVAS is the exception. The Grid is the only thing that ever talks to gvmd, so asking you to curate a password you'll never see is a tax. On first setup the backend rolls a 32-character URL-safe secret (~190 bits of entropy), persists it to config.yml, and feeds it to the container. The Settings panel shows a Managed automatically pill and a Rotate button — one click and the flow below runs with a freshly generated password. You never see it, type it, or lose it.

Gateway Integrations

A card below the core integrations, dedicated to the boxes that sit at the edge of the network — the ones doing firewalling and IDS. Two subsections, one per vendor, each independently enable-able. Configure neither and the Security page sits empty but discoverable. Configure one, the other, or both, and the alarm feed wakes up.

OPNsense. URL, API key, API secret — generated under System → Access → Users → API keys on the firewall itself. Verify SSL defaults off so the typical self-signed deployment works out of the gate; tighten it when you have a proper cert. IDS enabled is the switch that makes the scheduler walk the Suricata alerts endpoint on each poll — flip it off if you just want interface and lease data. Poll interval is seconds, default 60. Hit Test and the backend opens a session, reads firmware version and hostname, and hands the result back as proof the credentials resolve.

Firewalla. Two modes, one toggle. MSP (recommended) talks to Firewalla's cloud API — enter the MSP domain (e.g. mydomain.firewalla.net) and a Personal Access Token minted from the MSP dashboard. Local is the experimental path for users who only want to talk to the box directly on the LAN; the Firewalla local API is undocumented and volatile, so the card carries a warning and the integration ships as scaffold-only — Test succeeds but the poll returns empty snapshots until the schema solidifies. Alarms enabled gates the alarm fetch independently of device enrichment. Poll interval is seconds, default 120 — Firewalla's cloud API isn't latency-critical and aggressive polling risks rate limiting.

Both cards use the same secret-preservation rule as the core integrations: blank the API secret or token field, hit save, and the stored value stays put. The UI shows (unchanged) as a placeholder so it's visually obvious the old value is still in play.

Rotate OpenVAS Admin Password

The immauss/openvas image reads PASSWORD from the environment exactly once — the first time it boots against a fresh volume. After that, the env var is inert. Rotation means giving the image a clean slate with a fresh password baked into PASSWORD. Since the password is auto-managed, you don't type anything — just confirm you want to cycle it.

The escape hatch lives under the OpenVAS panel: Rotate. One button, one confirmation. The Grid generates a new secret, warns you about the wipe, and runs the flow:

  1. Snapshot & Stop
    The backend inspects homelab-openvas, captures the full container config — image, env, ports, labels, networks, restart policy, DNS aliases — then stops and removes the container.
  2. Wipe the Volume
    Removes the named data volume (homelab-dashboard_openvas-data). NVT feed cache, scan history, stored reports — all gone. The only way to get PASSWORD re-read is to give the image a clean slate.
  3. Recreate
    New container, same name, same everything — except USERNAME and PASSWORD are set to the freshly generated secret. Network aliases (openvas, homelab-openvas) are re-applied so DNS still resolves after the reset.
  4. Warmup & Verify
    First-boot NVT sync takes 5–10 minutes. The modal tracks a progress bar driven by the openvas:reset socket channel, polling authenticate every 15 seconds until gvmd accepts the new password. 15-minute timeout — on the other side of it, either success or a clean error.
  5. Persist
    Only after gvmd has authenticated with the new password does the Grid write it to config.yml. A failed reset leaves your stored password untouched — whatever was working before still works.

The whole flow writes reset_openvas_start, reset_openvas_success, or reset_openvas_failed rows into the Audit Log. You can close the modal and the reset keeps running in the background; the status pills keep ticking until ready.

⚠
This is destructive. Scan history, vulnerability reports, and the NVT feed cache are all wiped. Vulnerability findings already persisted to our SQLite database survive — the Vulnerabilities page keeps its history. But anything inside the OpenVAS volume itself is gone. Use this when authentication is genuinely broken, not as a routine password change.
◈
Requires docker socket access from inside the backend container. docker-compose.yml mounts /var/run/docker.sock read-write and adds group_add: [${DOCKER_GID}]. Set DOCKER_GID in your .env to your host's docker group id (getent group docker) — typically 999 on Debian/Ubuntu, 988 on RHEL/Fedora.
Scan Credentials

The second layer of authentication. OPENVAS_USER / OPENVAS_PASSWORD above is how the Grid talks to the scanner. Scan Credentials are how the scanner talks to your devices. Drop an SSH login here — password or private key — keyed to a target IP, and OpenVAS performs a credentialed scan. A proper interior walk-through rather than a port-knock from the outside.

The Auth Type dropdown flips the row between password and key. Pick key and the form swaps to a PEM textarea plus an optional passphrase field — paste the full private key (OpenSSH or PKCS#8, whatever your agent emits) and the Grid wires it straight into the GMP USERNAME_SSH_KEY credential object. Passwords land as USERNAME_PASSWORD. Same table, same UI — OpenVAS doesn't know the difference when the scan runs.

Use * as the target IP for a wildcard default applied to any device that doesn't have a more specific entry. You can also use CIDR ranges or dashed ranges (192.168.1.10-20) — Test Credentials expands them, fires an SSH auth probe at every resulting host, and drops a persistent badge next to the row. Green reads OK · N/M hosts. Red reads at least one credential failed and the detail — which IPs, which error — lands in the Audit Log for the post-mortem.

Each row is list / add / delete. No passwords or keys are ever shown back. The store lives in the local scan_credentials SQLite table (no sidecar, no vault, just plaintext inside the project's data volume).

⚠
Without a scan credential, OpenVAS sees exactly what an attacker on your LAN would — port banners and whatever the service greets strangers with. That's useful, but shallow. A credentialed scan reaches the package list, the kernel version, the real inventory. If you want deep findings, feed the Grid a login.
Scan Profiles
ProfileNmap FlagsUse When
quick -T4 -F Sprint — the top 100 ports, nothing else. Use it when you need to poll often and cheap.
standard -T4 -sV The default gait. Every port, with version detection. What the scheduler runs unless told otherwise.
deep -T3 -sV -O -A Full X-ray. OS fingerprinting, aggressive detection, the works. Slow — but this is what gives Claude something to read.

Data Flow — Network Scan

Nmap scan → merge_nmap_result() upserts device records in SQLite → build_topology_graph() constructs edges → topology:updated socket event → React Query invalidates → React Flow re-renders graph. Unknown devices → run_analysis_for_unknown_devices() → Claude pipeline (async) → claude:staged socket event → notification banner.

Data Flow — Vulnerability Scan

openvas_scan_job iterates online devices → run_openvas_scan(device_id, ip) → GMP session: create credential → create target → start task → poll → parse XML report → INSERT OR IGNORE INTO vuln_results → vuln:updated socket event → VulnsPage invalidates queries → CVE rows render.

Data Flow — Latency

latency_poll_job selects gateway + infrastructure devices → concurrent TCP connects on port 443 → 80 → 22 → 53 (timeout 2s) → RTT written to latency_samples → samples older than 24 hours pruned. The LatencyChart component polls GET /api/network/latency every 30 seconds and redraws its SVG path.

Undeploy / De-Rez

Sometimes the Grid has to come down. A clean slate. A quiet host. This is the process for that — a surgical teardown that de-rezzes the dashboard stack and leaves every other program on the box untouched.

Running the Script

Same host-resolution rules as deploy.sh — positional, env vars, or interactive prompt. The script never guesses.

terminal
# Option A — positional
$ ./undeploy.sh ubuntu@192.168.1.50

# Option B — environment variables
$ DEPLOY_HOST=192.168.1.50 DEPLOY_USER=ubuntu ./undeploy.sh

# Option C — interactive
$ ./undeploy.sh

# Skip the confirmation (destructive — know what you're doing)
$ ./undeploy.sh ubuntu@192.168.1.50 --yes

# Also purge upstream images (openvas, npm)
$ ./undeploy.sh ubuntu@192.168.1.50 --purge-images
What Gets De-Rezzed

Before anything is torn down, the script surveys the host and shows you exactly what it's about to erase. You'll get a preview, a warning, and a prompt to type YES before any destruction begins.

TargetAction
homelab-* containers Stopped and removed. Every container prefixed homelab-, including any left over from renamed project dirs.
Project volumes homelab-dashboard_* volumes — destroyed. Database, NPM state, OpenVAS feed cache. Gone.
Built images homelab-dashboard-frontend and homelab-dashboard-backend images are removed. Layer cache freed.
Upstream images Left in place by default (they're reusable). Pass --purge-images to also remove immauss/openvas and jc21/nginx-proxy-manager.
Deploy directory $HOME/homelab-dashboard (or DEPLOY_DIR) — recursively removed. Compose file, config, local DB — all wiped.
Docker networks The project bridge is cleaned up via docker network prune. Other networks on the host are left alone.
What Stays Untouched
  • Every other Docker project on the host — their containers, volumes, and images are never named by this script.
  • System packages. No apt runs here. No user accounts modified.
  • The Docker daemon itself. It keeps running.
  • Your SSH config, firewall rules, and anything outside the deploy directory.
⚠
This is destructive by design. The volumes go with the containers — every device history, every CVE record, every staged Claude change that hasn't been applied. If any of that matters, back up the deploy directory first. The script will ask you once; when you type YES, the Grid goes dark.
